## Findings

### 1) Shutdown crash: undefined `pnl_tracker`
**Severity:** High  
**Files:** `src/main.py:410-422`

### Problem
The shutdown summary uses `pnl_tracker`, but that variable is never defined in `run_bot()`.

### Evidence
Per-asset trackers are created here:
- `src/main.py:263-278`

But shutdown uses a different variable here:
- `src/main.py:410`
- `src/main.py:414`
- `src/main.py:422`

### Impact
On shutdown, the bot can raise `NameError` instead of printing the final session summary.

### Recommended fix
Either:
- aggregate all per-asset trackers into a session-level tracker, or
- build the final report by summing across `cyclers[i].pnl`.

### 2) Rejected/failed order can block future re-quote attempts
**Severity:** High  
**Files:** `src/execution/order_manager.py:91-111`

### Problem
`OrderManager.update_quotes()` stores `yes_price/no_price` and `yes_size/no_size` even when `_place_buy()` returns `None`.

### Why this is wrong
A `None` order ID means placement failed or was rejected (for example, post-only rejection). But local state is still updated as if the quote exists.

### Evidence
Relevant code:
- `src/execution/order_manager.py:92-100`
- `src/execution/order_manager.py:103-111`

The code does:
- attempt placement
- assign `active.yes_order_id = order_id`
- still assign `active.yes_price = quotes.yes_buy_price`
- still assign `active.yes_size = quotes.yes_buy_size`

Same issue exists for NO quotes.

### Repro result
A small local repro showed:
- first update marked as updated
- `order_id` remained `None`
- second identical update returned `False`
- bot stopped retrying despite no live order existing

### Impact
A temporary placement failure can leave the bot with no actual quote on the book while it believes it is quoting.

### Recommended fix
Only update `active.*price`, `active.*size`, and `updated=True` if `order_id` is truthy.

### 3) Live fill polling is throttled globally across all assets
**Severity:** High  
**Files:** `src/main.py:239-240`, `src/execution/clob_client.py:35`, `src/execution/clob_client.py:160-164`

### Problem
In live mode, all assets share the same `ClobClientWrapper` instance:
- `src/main.py:239-240`

But fill polling uses a single timestamp throttle:
- `src/execution/clob_client.py:35`
- `src/execution/clob_client.py:160-164`

### Why this is wrong
If BTC checks fills first, ETH/SOL/XRP checks within the next 2 seconds are skipped entirely.

### Repro result
A local repro calling `get_fills('market-A')` then `get_fills('market-B')` immediately produced only **one** actual client call.

### Impact
Fills for later assets may be delayed or missed during active trading, causing:
- stale inventory
- stale risk state
- stale P&L
- delayed re-quoting

### Recommended fix
Throttle per market, not globally. Example:
- replace `_last_fill_check_ts: float` with `_last_fill_check_ts_by_market: dict[str, float]`

### 4) Filled live orders remain in local `open_orders`
**Severity:** High  
**Files:** `src/execution/clob_client.py:176-208`

### Problem
`process_fills()` converts fills into normalized fill records, but does not remove or reduce the corresponding order from `self.open_orders`.

### Evidence
Relevant code:
- `src/execution/clob_client.py:191-207`

The function reads order context from `self.open_orders`, but never clears it afterward.

### Impact
The bot can think a filled order is still live. That can break:
- re-quote logic
- inventory/order synchronization
- active-order bookkeeping

### Recommended fix
After processing a fill:
- remove fully filled orders from `self.open_orders`, or
- track remaining size and clear once exhausted.

### 5) Settlement count is double-counted
**Severity:** Medium / High  
**Files:** `src/orchestration/market_cycler.py:299`, `src/monitoring/pnl_tracker.py:181-186`

### Problem
`record_settlement()` already increments `markets_settled`, and `MarketCycler._settle_market()` increments it again.

### Evidence
- `src/monitoring/pnl_tracker.py:181-186`
- `src/orchestration/market_cycler.py:299`

### Impact
Reported settlement counts can be inflated, producing misleading reporting and dashboard stats.

### Recommended fix
Have exactly one owner for this counter. Best option: keep increment logic inside `PnLTracker.record_settlement()` only.

### 6) End-of-market settlement does not fully clear inventory state
**Severity:** High  
**Files:** `src/orchestration/market_cycler.py:229-304`, `src/strategy/inventory.py:225-231`

### Problem
At settlement, the bot merges/redeems/logs, but does not clearly remove the market from inventory state.

`InventoryManager.settle_market()` exists but does not appear to be used in the end-of-market flow.

### Evidence
Settlement path:
- `src/orchestration/market_cycler.py:229-304`

Unused inventory settlement helper:
- `src/strategy/inventory.py:225-231`

### Impact
Old positions may linger in memory after settlement, which can cause:
- duplicate merge attempts
- stale exposure in future checks
- incorrect capital/exposure reporting

### Recommended fix
After settlement is completed, explicitly finalize/remove the market position from `InventoryManager.positions`.

### 7) Live merge path does not restore capital in PnL tracker
**Severity:** Medium  
**Files:** `src/execution/ctf_ops.py:506-527`, `src/execution/ctf_ops.py:924-927`

### Problem
In the live `BalanceMonitor.check_and_merge()` path, merge profit is recorded, but capital recovery is not added back to `pnl_tracker.current_capital`.

### Evidence
Live path records settlement only:
- `src/execution/ctf_ops.py:510-514`

Simulated path also restores capital:
- `src/execution/ctf_ops.py:924-927`

### Impact
Displayed/tracked available capital can drift below reality in live mode.

### Recommended fix

- `pnl_tracker.record_capital_recovery(usdc_recovery)`

### 8) NO-side quote protection derives from YES book instead of actual NO book
**Severity:** Medium  
**Files:** `src/orchestration/market_cycler.py:645-651`

### Problem
The code fetches only the UP token order book and infers the NO-side ask with:
- `best_ask_no = 1.0 - book_up.best_bid`

### Why this is risky
That is an approximation, not the real NO order book. It can be wrong when books are asymmetric or stale.

### Impact
Possible effects:
- avoidable post-only rejections
- weaker NO-side quotes
- incorrect anti-crossing logic

### Recommended fix
Fetch both token books independently:
- actual UP token book
- actual DOWN token book

### Evidence
